# PRD - MeetFlow AI

## 1. 문서 목적
이 문서는 "회의록이 실행으로"라는 비전을 가진 MeetFlow AI SaaS의 제품 요구 사항을 정의한다. 기획.md에 요약된 목적(회의 정리 자동화, 실행 관리 강화, 협업 비용 절감)을 구체적인 기능, 시나리오, 일정으로 전환해 제품/개발팀 간 공통 언어를 만든다.

## 2. 제품 비전 및 문제 정의
- 비전: "회의는 끝났지만, 실행은 자동으로 시작된다". 회의 데이터를 자동으로 전사, 요약, 실행 항목으로 변환하고 조직에게 필요한 알림을 제공해 실행에 집중하게 한다.
- 고객 문제
  - 회의 후 수작업 정리 시간이 많이 든다.
  - 실행 항목 누락·책임 미지정으로 업무가 지연된다.
  - 회의 내용을 적시에 공유하지 못해 협업 비용이 증가한다.
- 해결 방안: 음성/텍스트 입력을 자동으로 분석해 요약·액션을 생성하고 이메일, 다이어그램 등으로 즉시 배포함으로써 실행력과 투명성을 높인다.

## 3. 대상 고객 및 페르소나
1. **프로젝트 매니저(기업)**: 주간 프로젝트 회의를 주관, 후속 조치 미흡으로 일정 지연을 경험. 빠른 회의록과 액션 추적이 필요.
2. **린 스타트업 팀 리더**: 리소스 부족으로 회의 정리 시간이 부족. 자동화된 요약/액션 공유를 원함.
3. **대학 연구/학회 운영진**: 학회 회의나 세미나 내용을 구성원에게 공유해야 함. 이메일 자동 배포와 시각 자료가 필요.

## 4. 사용자 시나리오 요약
1. 사용자가 음성 녹음 파일을 업로드하거나 텍스트 회의록을 붙여넣는다.
2. 시스템이 전사를 수행하고 요약·Action Item을 생성한다.
3. 사용자는 추출 결과를 검토/수정 후 확정한다.
4. 확정된 요약/Action Item이 멤버에게 이메일로 발송되고, 요약 다이어그램이 생성돼 링크 또는 첨부로 공유된다.
5. 관리자 대시보드에서 회의별 실행 현황을 추적한다.(후속 단계)

## 5. 기능 요구사항
### 5.1 입력 및 전사
- [필수] 업로드 지원: 음성 파일(MP3, WAV 등)과 텍스트 문서(PDF, DOCX, TXT, 클립보드) 입력.
- [필수] 실시간 입력: 노트북·스마트폰 마이크를 이용한 브라우저 녹음/전사 지원, 녹음 중 일시정지·재개 기능.
- [필수] 전사 파이프라인: Whisper 등 STT 모델을 활용해 한국어/영어 전사, 발언자 구분(선택적).
- [필수] 상태 표시: 업로드/처리 진행률, 실패 시 재시도 안내.

### 5.2 회의 요약 생성
- [필수] 전사 결과를 기반으로 요약(개요, 결정 사항, 논의 포인트) 자동 작성.
- [선택] 요약 템플릿(회의 목적별) 제공.
- [필수] 사용자가 각 섹션을 편집 후 저장 가능.

### 5.3 Action Item 추출
- [필수] 책임자, 마감일, 업무 내용을 구조화된 형태로 추출.
- [필수] Confidence 점수 및 추천 마감일 제안.
- [선택] 사용자가 직접 추가/삭제/수정 가능한 UI.

### 5.4 회의 요약 다이어그램 (MVP)
- [필수-MVP] 요약 내용을 시각화한 다이어그램/타임라인 자동 생성 후 즉시 공유 가능하도록 한다.
- [선택] PNG/PDF 다운로드, 공유 링크 제공(공유 링크는 Supabase Storage 기반).

## 6. 비기능 요구사항
- **성능**: 60분 회의 파일 처리(전사+요약) 총 5분 이내, 처리 상태 지속 표시.
- **가용성**: 99.5% 월간 가용성, 실패 시 재시도 큐.
- **확장성**: 비동기 작업 큐(예: Celery, Sidekiq)와 스토리지(S3 호환) 분리.
- **보안/컴플라이언스**: 데이터 보관기간 설정(예: 기본 90일), 역할 기반 접근 제어, 이메일 수신자 화이트리스트.

## 7. 데이터 흐름/시스템 아키텍처(초안)
1. 클라이언트 → 업로드 API → 오브젝트 스토리지.
2. 워커가 STT 모델 호출 → 전사 결과 DB 저장.
3. LLM 요약/Action Item 추출 서비스 → 결과 저장.
4. 다이어그램 생성/공유 서비스 → Supabase Storage·Edge Function → 공유 링크 제공.
5. 프론트엔드 대시보드에서 회의별 상태 및 결과 조회.

## 8. 성공 지표(KPI)
- 회의 처리 소요시간(업로드~이메일 전송) 중간값 5분 이하.
- 자동 생성된 Action Item의 사용자 수정률 30% 이하.
- 주간 활성 회의 세션 수(MAU), 이메일 오픈율 60% 이상.
- 관리자 대시보드에서 완료 체크된 Action Item 비율 70% 이상.

## 9. 출시 범위 및 단계
- **MVP(1개월)**: 브라우저 녹음/파일 업로드 → 외부 STT API 연동 → 요약/Action Item 추출 → Supabase 저장 및 공유 링크/다운로드 제공. Vercel 기반 기본 UI/대시보드.
- **Beta(2~3개월)**: 이메일/Slack 알림 자동화, 다이어그램 생성 고도화, 사용자 편집 편의 기능, 기본 팀 관리.
- **GA(4개월 이후)**: 협업툴 심화 연동, 고급 보안/감사, 멀티 언어 지원, SLA 기반 인프라 확장.

## 10. 리스크 및 완화 전략
- STT/LLM 품질 편차 → 사용자 피드백 기반 재학습, 모델 버전 관리.
- 이메일 스팸 처리 → 발송 도메인 인증(SPF/DKIM) 및 발송량 모니터링.
- 개인정보 포함 데이터 저장 → 최소 데이터 수집, 자동 삭제 정책.
- 고용량 음성 파일 업로드 실패 → 청크 업로드 및 네트워크 재시도.

## 11. 오픈 이슈
1. 권한 모델(개인 vs 팀) 세부 정책 필요.
2. 다이어그램 생성 방식(템플릿 vs 자동 시각화 라이브러리) 확정 필요.
3. SLA/지원 범위에 따라 인프라 규모 결정 필요.

## 12. MVP 기술 스택 및 상세 계획
### 12.1 배포/인프라
- 프론트+백엔드: Next.js/Vercel(무료)로 정적 페이지+Serverless Functions 구성. Functions는 업로드/전사 요청 프록시, 요약/액션 추출, Supabase 호출을 담당한다.
- 데이터/인증: Supabase 무료 플랜으로 사용자 인증, Postgres DB, 오브젝트 스토리지를 사용한다. 액션 아이템/요약/파일 메타데이터 저장.
- STT/LLM: OpenAI Whisper API 또는 AssemblyAI 등 외부 STT API를 호출하고, GPT-4o-mini 수준 모델로 요약/액션 추출. 장시간 전사는 비동기 Polling 패턴으로 처리한다.
- 이메일: MVP에서는 미구현. Beta 단계에서 Resend/SendGrid와 연동해 Vercel Edge Function이나 Supabase Function에서 발송한다.

### 12.2 구현 단계(주차 단위)
1) 환경 세팅(Week 1): Vercel 프로젝트/도메인 연결, Supabase 프로젝트 생성, 환경 변수/비밀키 세팅.
2) 입력 파이프라인(Week 1~2): 브라우저 녹음 UI, 파일 업로드 컴포넌트, Vercel API에서 Supabase Storage 업로드 후 STT API 호출.
3) 요약·액션 엔진(Week 2): STT 결과를 기반으로 LLM 프롬프트 설계, 결과를 Supabase DB에 저장, 편집 UI 구현.
4) 리뷰/공유(Week 3): 사용자가 결과를 확인·수정하고, 공유 링크/다운로드(임시 이메일 대안) 제공.
5) 다이어그램/대시보드(Week 4): LLM 출력 → Mermaid/Chart.js로 시각화, 회의별 리스트/상태 페이지.
6) 안정화(Week 4): 모니터링, 에러 로깅(Supabase Edge Functions 또는 Sentry) 추가.

### 12.3 기술 고려 사항
- Vercel 함수 시간 제한을 넘지 않도록 STT/LLM 호출은 비동기 Webhook 또는 폴링으로 설계하고, 긴 작업은 Supabase에 상태 저장 후 클라이언트에서 주기적 조회.
- Supabase RLS 정책을 활용해 팀/사용자 데이터 분리를 초기부터 설계한다.
- 파일 저장비 절약을 위해 전사 완료 후 음성 파일 자동 만료(예: 30일 후 삭제) 배치 작업을 Supabase Scheduled Function으로 구성.
- 베타 단계에서 이메일 기능을 붙일 수 있도록 공용 알림 인터페이스(예: NotificationService) 추상화.

위 요구사항을 기준으로 디자인, 기술 아키텍처, 일정 산출을 진행한다. 구현 중 변동 사항은 본 문서를 업데이트해 추적한다.
